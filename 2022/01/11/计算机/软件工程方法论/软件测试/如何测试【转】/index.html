在本文中，我们会以一个简单的上传、下载文件的服务来作为例子。在这个例子中：

  • 有一个存储引擎层（storage_engine），可以把文件放在本地内存、本地磁盘或是HDFS上等等。

如何开展测试？

在我们写代码的时候，或者对他人代码进行测试的时候，我们应该怎么开展一个测试工作？这里给出一些建议：

设计阶段

工作内容

给出测试计划，需要包括：

  • 功能测试：这些测试定义了该项工作的各个功能点
  • 专项测试：该项工作中一些大的风险点，复杂点可能需要专门的测试

说明

测试计划的目的在于

  • 促使你能够系统的、TopDown的思考问题
  • 促使你关注该项任务的最初目标、核心风险
  • 帮助你清晰的定义产品功能

一些Tips：

  • 测试计划中大多数用例不需要详细的描述

举例

  • 功能测试
    • 能够正确的上传、下载、删除、枚举
    • 能够正确的处理并发请求
    • 能够处理超大文件（>5GB）
  • 专项测试
    • 性能测试
    • 错误注入测试

编码阶段

工作内容

  • 及时记录感受到的复杂点、风险点，并转化为测试用例

说明

在编码过程中，常常会发现此前没有考虑到的一些细节和风险，这些都要转化为测试用例

举例

  • 正在下载中的文件，即使被删除了，也能继续下载

验收阶段

工作内容

  • 交叉测试

说明

交叉测试的好处非常多（多一个人理解系统，多一个小黄鸭），建议大家经常性使用这一招。

举例

  • 有些工作一定要另外一个同学进行“交叉测试”, 一些参考
    • 复杂的逻辑，如分布式系统中数据均衡
    • 高风险的工作，如数据的删除、落盘格式的改变
    • 对外接口，如本文示例中的HTTP API

全过程

  • 回归测试

在我们开发、测试、灰度过程中，遇到的任何一个Bug，都要变成一个回归测试。

总结

相比对某个专门函数、类或是某个特定逻辑层的测试有如下缺点：

  • 代码经常会发生重构，而重构会使得这类测试被作废了，或要做很大的改动

Rule 4：测试代码是一等公民：需要充分的注释和抽象，避免拷贝、粘贴代码

测试代码通常被认为是二等公民，表现在：

  • 大部分Review代码的人并不会去认真Review测试用例
  • 测试代码通常是繁琐、冗长的
  • 测试代码本身是没有测试的

而实际上，我们要认识到，测试代码是我们最重要的资产，重要程度超过代码本身。所以我们需要：

  • 每个测试用例都要给出注释，说明，我要测试什么，我要验证什么，我是如何测试的
  • 一些通用的部分，一定要抽象出来，避免拷贝粘贴

关于随机性

一些Tips：

  1. 大部分的测试用例是不需要注入随机性的。因为大部分情况下，我们需要的是确定性的测试用例。
  2. 对于一次测试任务，我们可以给出1到2个随机性测试用例。类似于fuzzing test的。
  3. 随机故障测试是非常强大的工具，如端到端的随机I/O出错、随机Kill测试。

如何Review测试？

测试代码通常是枯燥无味的，大部分reviewer都很难对测试代码提起兴趣。结果就是，测试代码得不到充分的Review，经常出现测试不充分，或者测试代码压根就没有测到点子上。

这里列一些Tips来帮助我们Review测试代码：

  • 比对测试计划：Reviewer根据设计文档、功能需求，自己先列一个测试计划，然后和被测试代码的测试计划进行比对
  • 关注测试代码的可读性：是不是有足够的注释说明测试的目的、测试的步骤、验证的手段？是不是有足够的抽象，避免了重复的代码，关注的是测试逻辑，而不是测试的实现？
  • 重点阅读一些核心用例：至少要深入的、仔细的阅读几个核心用例，看看测试人员的测试方法、测试代码是否正确。
  • 关注测试真实性：
    • 一个测试有多少是Mock的东西？Mock的东西越多，测试的真实性、有效性就越低。
    • UT的真实性不如E2E。
  • Review代码时，收集代码中的复杂点、风险点，确认这些地方被测试覆盖了。
